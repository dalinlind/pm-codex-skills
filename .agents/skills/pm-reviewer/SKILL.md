# PM Reviewer

## Purpose

对 PRD、原型或功能方案进行多角色评审，找出逻辑漏洞、范围不清、规则缺失、数据口径不明、实现风险和上线风险，并给出可执行修改建议。

## When to Use

- 用户要求评审、审查、挑刺、找问题、检查 PRD 或原型。
- 用户希望从研发、测试、设计、运营、销售、客户成功等角度看需求。
- 需求涉及复杂权限、状态流转、AI 判断、数据看板、企业客户交付或跨系统协作。

## Input

| 信息 | 说明 |
| --- | --- |
| 被评审材料 | PRD、原型、流程图、页面描述、接口说明或上线方案 |
| 评审目标 | 找风险、补规则、控范围、准备评审会或上线前检查 |
| 业务背景 | 产品类型、客户类型、使用场景 |
| 版本范围 | 本期范围和非本期范围 |

## Output Structure

默认输出：

1. 总体结论
2. 高风险问题
3. 分角色评审意见
4. 需求完整性检查
5. 流程与状态检查
6. 数据与埋点检查
7. AI 能力风险，如适用
8. 上线与运营风险
9. 修改建议优先级
10. 待确认问题

## Rules

- 先给结论，再列问题。
- 问题要具体，不能只说“需要补充规则”。
- 每个问题必须说明影响和建议。
- 优先识别会影响研发估时、测试覆盖、用户使用、客户交付、数据统计或合规的风险。
- 对 B 端需求重点检查权限、组织、角色、数据隔离、配置、审批、审计、批量操作和异常回滚。
- 对 AI 功能重点检查误判、不可解释、置信度、人工复核、训练/推理数据边界、模型失败兜底和用户申诉。
- 对招聘/HR 场景重点检查职位、候选人、简历、面试、评价、录用、企业账号、通知触达和数据同步。
- 不只批评，要给出可落地的改法。
